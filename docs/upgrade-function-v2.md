## 목표

이번 작업의 목표는 다음과 같다.

1. CLI 사용자가 기간을 명시적으로 선택하거나 입력할 수 있게 정리한다.
2. 사용자가 정확한 `projectKey`를 모르더라도 프로젝트 명칭 일부로 조회를 시작할 수 있게 한다.
3. 잘못된 기간값, 잘못된 프로젝트 입력에 대해 사용자가 이해 가능한 오류 메시지를 제공한다.

## 작업 범위

이번 작업에서는 아래 항목만 구현 대상으로 본다.

### 1. 기간 입력 정교화

현재 `period`는 입력은 받지만 사실상 `this_week`만 의미 있게 동작한다.

이번 작업에서는 아래 기간값을 우선 지원 대상으로 설계 및 구현한다.

1. `this_week`
2. `last_week`
3. `today`
4. `yesterday`
5. `custom_range`

추가 원칙은 다음과 같다.

1. `custom_range`는 `startDate`, `endDate` 입력을 추가로 받아 처리한다.
2. CLI는 자유 문자열 1회 입력보다 명시 선택형 흐름을 우선한다.
3. 서버는 지원하지 않는 기간값을 그대로 통과시키지 말고 검증 오류로 처리한다.
4. 기간값은 공통 `Query Schema` 기준으로 정규화한다.
5. JQL 생성 시 각 기간값이 실제 조회 조건으로 반영되어야 한다.

### 2. 프로젝트 부분명 검색 지원

현재는 `projectKey` 정확 입력만 가능한 구조다.

이번 작업에서는 아래 방향으로 확장한다.

1. 사용자가 프로젝트 키를 모르고 이름 일부만 아는 경우를 지원한다.
2. Jira 프로젝트 식별용 lookup 단계를 추가한다.
3. 부분 일치 방식은 MySQL `LIKE`와 유사한 `contains` 개념으로 처리하되, 실제 구현은 Jira API 기반 조회로 처리한다.
4. 조회 결과가 1건이면 자동 선택한다.
5. 조회 결과가 여러 건이면 후보 목록을 보여주고 사용자가 선택하게 한다.
6. 조회 결과가 0건이면 이해 가능한 안내 메시지를 출력한다.
7. 기존 `projectKey` 직접 입력 흐름은 유지한다.

### 3. API/CLI 오류 메시지 개선

이번 작업에서는 아래 상황에 대한 메시지를 구분해서 제공한다.

1. 기간 입력 오류
2. 프로젝트 lookup 실패
3. 프로젝트 후보 다건 조회
4. 잘못된 프로젝트 키 입력

메시지는 사용자가 다음에 무엇을 입력해야 하는지 알 수 있게 작성한다.

## 검증 항목

완료 전 아래 항목을 검증해야 한다.

1. `this_week`, `last_week`, `today`, `yesterday`가 실제 JQL에 반영된다.
2. `custom_range` 입력 시 시작일과 종료일이 검증된다.
3. 잘못된 기간 입력 시 명확한 오류 메시지가 나온다.
4. 프로젝트 키 직접 입력 조회가 기존처럼 동작한다.
5. 프로젝트명 일부 입력 시 후보 1건, 다건, 0건 케이스가 각각 처리된다.
6. `assignee`, `project`, `assignee_project` 세 모드 모두 기존 흐름이 깨지지 않는다.
7. Markdown 저장 흐름이 유지된다.
8. 검증 결과를 완료 보고에 포함한다.
