# 작업 지시서: 주간 조회 기준 개선

## 문서 목적

이 문서는 주간 Jira 조회 결과 품질을 개선하기 위한 작업 지시서다.

현재 주간 조회는 `updated` 기준으로 이슈를 검색하고 있다. 이 방식은 최근에 실제 작업하지 않은 이슈라도 자동화, 메타데이터 변경, 상태 이력, 부수 수정 등의 이유로 다시 조회 결과에 포함될 수 있다.

이번 작업은 사용자 입력 단계를 늘리지 않고, 내부 조회 기준만 조정하여 `this_week`, `last_week` 조회 결과의 과포함 문제를 줄이는 것을 목표로 한다.

## 작업명

`주간 조회 기준 개선 - 사용자 입력 유지, 내부 JQL 기준 보정`

## 목표

이번 작업의 목표는 다음과 같다.

1. CLI 단계 수를 늘리지 않는다.
2. 사용자는 기존처럼 조회 모드와 기간만 선택한다.
3. `this_week`, `last_week` 조회 결과에서 오래된 이슈가 단순 갱신으로 다시 섞이는 문제를 줄인다.
4. 주간 업무 취합 목적에 더 맞는 조회 결과를 제공한다.

## 작업 범위

이번 작업에서는 아래 항목만 구현 대상으로 본다.

### 1. 사용자 입력 구조 유지

사용자 입력은 기존 구조를 유지한다.

1. 조회 모드 선택
2. assignee 또는 project 입력
3. 기간 선택
4. Markdown 저장 여부 선택

이번 작업에서는 새로운 선택 항목이나 옵션을 추가하지 않는다.

### 2. 주간 조회 내부 기준 변경

현재 `this_week`, `last_week`는 `updated` 기준으로 처리되고 있다.

이번 작업에서는 아래 방향으로 내부 기준을 조정한다.

1. `this_week` 조회는 `updated` 단독 기준 대신 `status changed` 중심 기준으로 변경 검토
2. `last_week` 조회는 `updated` 단독 기준 대신 `status changed` 중심 기준으로 변경 검토
3. 주간 조회 결과가 실질적인 업무 변경 이력에 더 가깝도록 JQL을 재구성한다

### 3. 일간 조회와 범위 조회 영향 최소화

아래 항목은 기존 동작을 최대한 유지한다.

1. `today`
2. `yesterday`
3. `custom_range`

이번 작업은 주간 조회 품질 개선을 우선 목표로 하며, 일간 조회와 날짜 범위 직접 입력 기능은 회귀 없이 유지되어야 한다.

## 설계 원칙

이번 작업은 아래 원칙을 따른다.

1. 사용자 입력 단계는 늘리지 않는다.
2. Query Schema 변경은 최소화한다.
3. 조회 품질 개선은 CLI가 아니라 JQL 생성기 내부에서 처리한다.
4. 기존 `assignee`, `project`, `assignee_project` 세 모드는 유지한다.
5. Markdown 및 console 출력 구조는 유지한다.
6. 대규모 구조 개편 없이 현재 구조 안에서 해결한다.

## 비범위

아래 항목은 이번 작업 범위에서 제외한다.

1. 새로운 CLI 질문 단계 추가
2. 조회 기준 선택 옵션 추가
3. Web UI 변경
4. MCP 조회 기준 확장
5. changelog 기반 정밀 후처리 전면 도입
6. 다중 조건 고급 필터 추가
7. 자연어 기반 조회 조건 해석

## 권장 구현 방향

구현은 아래 순서를 우선 참고한다.

1. 현재 기간별 JQL 생성 로직을 점검한다.
2. `this_week`, `last_week`에서 사용하는 `updated` 기준을 대체할 내부 조건을 설계한다.
3. Jira JQL에서 사용할 수 있는 `status changed` 계열 조건을 적용한다.
4. `today`, `yesterday`, `custom_range` 기존 동작에 영향이 없는지 확인한다.
5. 결과 과포함이 실제로 줄어드는지 비교 검증한다.

## 검증 항목

완료 전 아래 항목을 검증해야 한다.

1. 사용자 입력 단계가 늘어나지 않는다.
2. `this_week`, `last_week` 조회 시 몇 주 전 작업 이슈가 단순 갱신 때문에 다시 포함되는 비율이 줄어든다.
3. `assignee`, `project`, `assignee_project` 세 모드가 모두 동작한다.
4. `today`, `yesterday`, `custom_range` 기존 동작이 깨지지 않는다.
5. `npm run typecheck`가 통과한다.
6. `npm run build`가 통과한다.
7. 가능하면 실제 Jira 데이터 기준으로 변경 전후 결과를 비교한다.

## 주의사항

이번 작업은 `updated` 기준의 과포함 문제를 줄이기 위한 1차 보정이다.

`status changed` 기준은 주간 취합 정확도를 높일 수 있지만, 본문 수정이나 설명 수정처럼 상태 변경 없이 수행한 작업은 일부 누락될 수 있다. 따라서 이번 단계는 주간 조회 품질 개선을 우선하고, 필요 시 후속 단계에서 changelog 기반 정밀화 여부를 검토한다.
