# 작업 지시서: v0.3.0 npm 배포 준비

## 1. 목적

이 문서는 `qwen-jira-mcp` 패키지를 `npm` 배포 직전 상태로 정리하기 위한 작업 범위를 고정한다.

이번 작업의 목표는 실제 `npm publish` 실행이 아니라, 패키징 산출물과 메타데이터를 점검하고 공개 배포 전 필요한 정리를 반영하는 것이다.

## 2. 배경

현재 프로젝트는 CLI, 로컬 API 서버, MCP 실행 진입점을 `npm` 패키지로 제공할 수 있는 구조를 갖고 있다.

하지만 배포 준비 관점에서 다음 항목을 점검할 필요가 있다.

1. `npm pack` 기준 실제 포함 파일이 적절한지
2. 불필요한 빌드 산출물이 패키지에 포함되지 않는지
3. `README.md`가 설치 후 사용 흐름을 정확히 설명하는지
4. `package.json` 메타데이터와 스크립트가 배포 전 점검 흐름에 맞는지
5. 실제 배포 전 남는 차단 사항이 있다면 무엇인지

## 3. 범위

이번 작업 범위는 아래로 제한한다.

1. `package.json`의 npm 배포 관련 메타데이터/스크립트 점검 및 정리
2. 패키지 포함 파일 정리
3. `README.md`의 설치/실행/주의사항 문구 정리
4. 필요 시 배포 준비용 보조 스크립트 추가
5. 배포 전 확인 결과를 문서 또는 README에 반영

## 4. 세부 요구사항

1. `npm pack --dry-run` 기준 패키지에 불필요한 파일이 포함되지 않도록 정리한다.
2. stale build 산출물 때문에 잘못된 파일이 패키지에 포함되지 않도록 빌드/패키징 스크립트를 정리한다.
3. `README.md`에서 실제 존재하지 않는 문서나 기능에 대한 잘못된 참조가 있으면 정리한다.
4. 사용자는 전역 설치 후 아래 흐름을 이해할 수 있어야 한다.
   1. 환경변수 또는 `.env` 준비
   2. `qwen-jira-server` 실행
   3. `qwen-jira` 실행
   4. 필요 시 `qwen-jira-server status|stop`
5. `MCP`는 현재 scaffold 상태임을 README에 명확히 유지한다.
6. 실제 `npm publish`는 수행하지 않는다.
7. 라이선스 정책은 임의로 변경하지 않는다.
8. 공개 배포 관점에서 남는 차단 사항이 있으면 결과 보고에 명시한다.

## 5. 제외 범위

이번 작업에서 아래는 하지 않는다.

1. 실제 `npm publish` 실행
2. npm 계정 로그인/권한 설정
3. 패키지 이름 변경
4. 새 기능 구현
5. Jira 기능 로직 변경
6. Web API, MCP 기능 확장
7. 법적 판단이 필요한 라이선스 선택/변경

## 6. 예상 변경 대상

1. `package.json`
2. `README.md`
3. 필요 시 `scripts/` 하위 보조 파일
4. 필요 시 패키징 관련 설정 파일

## 7. 산출물

1. npm 배포 준비가 반영된 패키지 설정
2. 최신 사용 흐름이 반영된 README
3. 패키징 점검 결과
4. 남은 배포 차단 사항 목록

## 8. 검수 기준

1. `npm run build`가 통과한다.
2. `npm pack --dry-run`가 통과한다.
3. 패키지에 의도하지 않은 stale build 파일이 포함되지 않는다.
4. README 설치/실행 설명이 현재 동작과 일치한다.
5. 실제로 존재하지 않는 문서 참조가 제거되거나 수정된다.
6. 라이선스처럼 사용자 결정이 필요한 항목은 임의로 바꾸지 않고 차단 사항으로 명시한다.

## 9. 보고 기준

최종 보고에는 아래를 포함한다.

1. 변경 파일 목록
2. `npm run build` 결과
3. `npm pack --dry-run` 결과 요약
4. 남은 차단 사항

## 10. 작업 체크리스트

1. `package.json`의 패키징 스크립트와 포함 파일 기준을 점검한다.
2. stale build 산출물이 패키지에 남지 않도록 빌드 전에 정리 흐름을 보장한다.
3. `README.md`의 설치/실행 흐름과 깨진 참조를 점검한다.
4. `npm run build`와 `npm pack --dry-run`으로 결과를 검증한다.
5. 라이선스 정책처럼 사용자 결정이 필요한 항목은 변경하지 않고 차단 사항으로 보고한다.
