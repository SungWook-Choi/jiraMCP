# 작업 지시서

## 문서 목적

이 문서는 `qwen3JiraMCP` 프로젝트의 초기 MVP 개발 범위와 우선순위를 고정하기 위한 작업 지시서다.

배포 전략은 `npm 배포`를 우선 방향으로 정리한다. `Docker 배포`와 외부 서비스 운영은 후속 범위로 둔다.

## 프로젝트 목표

이 프로젝트의 1차 목표는 다음 3가지다.

1. Jira 이슈를 조회하고 사람이 읽기 쉬운 형태로 정리한다.
2. 사용자가 터미널에서 직접 사용할 수 있는 `CLI`를 제공한다.
3. 에이전트가 호출할 수 있는 `MCP 서버` 인터페이스를 제공한다.

장기 목표는 다음을 포함한다.

1. 동일 코어를 사용하는 `Web UI` 연동 가능 구조 확보
2. 조회된 Jira 결과를 로컬 LLM으로 요약 또는 정리하는 확장 가능 구조 확보

## 핵심 원칙

1. Jira 조회는 `LLM`이 아니라 `Jira API`로 수행한다.
2. `LLM 연동`은 초기 MVP의 필수 조건이 아니다.
3. 동일한 비즈니스 로직을 `CLI`와 `MCP`가 함께 재사용해야 한다.
4. 사용자는 대화형 CLI 흐름으로 입력을 확인하면서 작업을 진행할 수 있어야 한다.
5. `[주의사항]` 같은 규칙 기반 정보 추출은 코드에서 처리한다.
6. 결과는 콘솔 출력이 기본이며, 필요 시 Markdown 저장 옵션을 제공한다.
7. 입력은 자유 자연어 해석보다 `명시적 조회 모드 선택`을 우선한다.
8. 조회 조건은 내부적으로 공통 Query Schema로 정규화한다.
9. 로컬 실행과 Docker 실행 모두에서 동일한 CLI 경험을 제공해야 한다.
10. 사용자의 Jira 연결은 실행 환경의 환경변수를 기준으로 처리한다.
11. 초기 런타임과 모듈 구조는 `NestJS`를 기준으로 설계한다.

## 추천 구현 방향

프로젝트는 `NestJS + TypeScript`를 기준으로 설계한다.

프로젝트는 아래 3계층으로 나눈다.

1. `core`
Jira API 호출, 기간 계산, 담당자 식별, 결과 정규화, 주의사항 추출, 출력 데이터 생성 담당

2. `cli`
사용자 입력, 질문/응답 흐름, 실행 옵션, Markdown 저장 담당

3. `mcp`
에이전트 호출용 tool 정의와 요청 매핑 담당

NestJS 기준 초기 모듈 방향은 다음을 따른다.

1. `JiraModule`
2. `QueryModule`
3. `SummaryModule`
4. `CliModule`
5. `McpModule`

후속 확장 시 아래 모듈을 추가할 수 있다.

1. `WebModule` 또는 `ApiModule`
2. `LlmModule`

## 실행 UX 방향

사용자는 특정 명령어를 입력해 `User Jira CLI`를 시작할 수 있어야 한다.

예시 실행 개념:

```bash
qwen-jira
```

위 명령이 실행되면 대화형 CLI 세션이 시작되고, Jira 조회 질문을 순차적으로 진행한다.

이 UX는 아래 환경에서 동일하게 유지되어야 한다.

1. 로컬 실행
2. Docker 실행

## 환경변수 기반 Jira 연결 원칙

Jira 인증 정보는 사용자가 실행한 환경의 환경변수를 기준으로 읽는다.

즉, CLI가 시작되면 현재 프로세스 환경에서 Jira 설정을 확인하고, 그 정보로 Jira API 연결을 시도한다.

이 원칙은 아래 인터페이스에 공통 적용한다.

1. CLI
2. MCP 서버
3. 추후 Web 서버

## 초기 조회 모드

초기 MVP는 아래 3가지 조회 모드를 지원 대상으로 본다.

1. `담당자 기준 조회`
- 입력: `담당자`, `기간`
- 예시 조건: 특정 담당자의 이번 주 업무

2. `프로젝트 기준 조회`
- 입력: `프로젝트 키`, `기간`
- 예시 조건: 특정 프로젝트의 이번 주 업무

3. `담당자 + 프로젝트 기준 조회`
- 입력: `담당자`, `프로젝트 키`, `기간`
- 예시 조건: 특정 프로젝트 안에서 특정 담당자의 이번 주 업무

초기 MVP에서는 위 3가지 모드만 명시적으로 지원한다.

## 보류하는 조회 확장

아래 조회 방식은 확장 후보로 열어두되 초기 MVP 범위에서는 제외한다.

1. 다중 담당자 동시 조회
2. 다중 프로젝트 동시 조회
3. `담당자: 전체`
4. Jira 사용자 목록을 먼저 불러온 뒤 CLI에서 선택하는 흐름

`담당자: 전체`는 의미 정의가 먼저 필요하다.
현재 기준으로는 `프로젝트 + 기간 내 이슈가 있는 assignee 전체를 그룹핑`하는 방식이 가장 현실적인 후속 확장안이다.

## 1차 사용자 흐름

초기 CLI는 아래 흐름을 기준으로 구현한다.

1. 사용자가 특정 명령어로 `User Jira CLI`를 실행한다.
2. CLI가 Jira 연결에 필요한 환경변수를 확인한다.
3. 환경변수가 없거나 불완전하면 명확한 오류 메시지를 출력한다.
4. CLI가 `조회 방식`을 묻는다.
5. 사용자는 아래 중 하나를 선택한다.
- `담당자 기준`
- `프로젝트 기준`
- `담당자 + 프로젝트 기준`
6. 선택한 조회 방식에 따라 필요한 입력값만 받는다.
7. `기간:`은 기본값을 `이번 주`로 둔다.
8. 서버가 입력값을 공통 Query Schema로 변환한다.
9. 서버가 Query Schema 기준으로 Jira API를 호출한다.
10. 결과를 규칙 기반으로 정리한다.
11. 정리 결과를 콘솔에 출력한다.
12. 선택 시 Markdown 파일로 저장한다.

## CLI 입력 규칙

대화형 CLI는 아래 입력 전략을 따른다.

1. 처음부터 한 줄 자연어를 해석하려 하지 않는다.
2. 먼저 `조회 모드`를 선택하게 한다.
3. 선택한 모드에 필요한 필드만 질문한다.
4. 사용자가 입력한 값은 모두 내부 Query Schema로 정규화한다.
5. 추후 비대화형 CLI 옵션이 추가되더라도 동일 Query Schema를 사용한다.

## Query Schema 초안

내부 코어와 MCP는 아래와 유사한 공통 입력 구조를 사용한다.

```json
{
  "mode": "assignee | project | assignee_project",
  "assignees": ["홍길동"],
  "projectKeys": ["ABC"],
  "period": "this_week",
  "output": {
    "format": "console | markdown"
  }
}
```

설계 원칙은 다음과 같다.

1. CLI 입력값은 모두 이 구조로 변환한다.
2. MCP tool 입력도 가능한 한 이 구조에 맞춘다.
3. 다중 담당자와 다중 프로젝트는 지금 당장 구현하지 않더라도 배열 구조를 유지해 확장 가능성을 남긴다.

## 다음 아키텍처 방향

현재 CLI는 직접 실행형 흐름으로 동작한다.
다음 단계에서는 `server-centered architecture`로 전환하는 것을 목표로 한다.

핵심 방향은 아래와 같다.

1. `NestJS Server`는 계속 실행되는 중심 서비스가 된다.
2. `User CLI`는 짧게 실행되는 클라이언트가 된다.
3. `MCP`는 같은 서버 코어를 재사용하는 어댑터가 된다.
4. `Web UI`는 나중에 같은 서버 API 위에 붙는다.

## Server-Centered 구조

### 1. NestJS Server

계속 실행되는 서버 프로세스다.

역할:

1. Jira 인증과 설정 관리
2. assignee lookup
3. QuerySchema 처리
4. JQL 생성
5. Jira API 호출
6. `[주의사항]` 추출
7. Markdown 렌더링
8. 추후 LLM 요약 확장

### 2. User CLI Client

사용자가 터미널에서 호출하는 짧은 실행 단위다.

역할:

1. 사용자 입력 수집
2. 로컬 서버 요청
3. 결과 콘솔 출력
4. 필요 시 Markdown 저장 요청

### 3. MCP Adapter

에이전트 호출용 인터페이스다.

역할:

1. MCP tool 입력 수집
2. 서버 기능 호출
3. 결과 반환

### 4. Web UI

나중에 붙는 브라우저 화면이다.

역할:

1. 서버 API 호출
2. 조회 결과 목록/상세 표시

## 권장 통신 방식

초기 전환 단계에서는 `localhost HTTP`를 권장한다.

1. `CLI -> Server`: localhost HTTP
2. `MCP -> Server`: localhost HTTP 또는 같은 코어 직접 호출
3. `Web -> Server`: HTTP API

## npm 배포 방향

이 프로젝트는 `npm 배포`를 우선 배포 전략으로 삼는다.

핵심 방향은 아래와 같다.

1. 사용자는 `npm install -g` 또는 `npx`로 CLI를 실행할 수 있어야 한다.
2. 같은 패키지 안에서 CLI, server, MCP 실행 경로를 함께 제공하는 방향을 우선 검토한다.
3. Jira 인증 정보는 패키지에 포함하지 않고, `.env` 또는 환경변수로만 주입한다.
4. 배포 가능한 패키지 구조와 사용자 실행 경험을 먼저 안정화한 뒤 publish를 진행한다.

## npm 배포 준비 항목

배포 전 아래 항목을 정리해야 한다.

1. 패키지 이름 확정
2. `bin` 명령 구조 확정
3. server 실행 명령과 CLI 실행 명령 정리
4. README 배포용 설치/실행 가이드 정리
5. `.env`, `node_modules`, `dist`, 개발 산출물 포함 정책 정리
6. `package.json`의 `files`, `bin`, `scripts` 검토
7. publish 전 최종 검증 절차 정리

## npm 배포 사용 경험 목표

사용자는 아래 흐름으로 접근할 수 있어야 한다.

1. `npm install -g <package-name>`
2. 환경변수 또는 `.env` 설정
3. server 실행
4. CLI 실행

예상 목표 예시는 아래와 같다.

```bash
qwen-jira-api
qwen-jira
```

또는 단일 패키지 내 `npm run`/`npx` 진입을 유지하면서 점진적으로 CLI 명령을 고정할 수 있다.
