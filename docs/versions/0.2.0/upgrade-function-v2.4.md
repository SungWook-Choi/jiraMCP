# 작업 지시서: 작업 이력 판정 정밀화 3차

## 문서 목적

이 문서는 Jira 조회 결과를 단순 기간/상태 기준이 아니라 실제 작업 이력 기준으로 더 정확하게 판정하기 위한 3차 작업 지시서다.

현재 조회는 `담당자 또는 프로젝트 + 기간` 조건으로 이슈를 조회한 뒤, 일부 기간에 대해 `created + status changed` 기준을 사용하고 있다. 이 방식은 기존 `updated` 기준보다 정확도가 높지만, 설명/요약/시간 추적처럼 상태 변경 없이 발생하는 작업 흔적은 1차 후보군에서 빠질 수 있다.

1. 설명/요약 수정
2. 시간 추적 수정
3. 담당자 변경
4. 댓글 변경 여부

같은 실제 작업 흔적을 충분히 반영하지 못한다.

이번 작업은 사용자 입력 구조는 그대로 유지하면서, 조회된 이슈의 `changelog`와 필요한 경우 `comment`를 확인해 “실제 작업한 이슈”만 더 정확하게 판정하는 것을 목표로 한다.

## 작업명

`작업 이력 판정 정밀화 3차 - changelog 기반 작업 필드 판정`

## 목표

이번 작업의 목표는 다음과 같다.

1. 사용자 입력 단계는 추가하지 않는다.
2. 기존 조회 조건은 `담당자 또는 프로젝트 + 기간` 구조를 유지한다.
3. 조회된 이슈에 대해 `changelog`를 확인하여 실제 작업 여부를 판정한다.
4. 아래 항목이 기간 내 변경된 경우 작업 이슈로 인정한다.
   - 상태 변경
   - 담당자 변경
   - 설명 변경
   - 요약 변경
   - 시간 추적 변경
5. 가능하면 댓글 변경도 후속 검토 대상으로 정리한다.

## 작업 범위

이번 작업에서는 아래 항목만 구현 대상으로 본다.

### 1. 1차 조회 구조 유지

기존 조회 방식은 그대로 유지한다.

1. 조회 모드 선택
2. assignee 또는 project 입력
3. 기간 선택
4. Markdown 저장 여부 선택

1차 조회는 기존 입력 구조를 유지하되, 기간 내 `updated`, `created`, `status changed`를 모두 후보군에 포함할 수 있도록 Jira 이슈 후보를 가져온다.

### 2. 2차 작업 판정 추가

조회된 이슈 목록에 대해 2차 판정 단계를 추가한다.

1. 각 이슈의 changelog를 조회한다.
2. 기간 내 변경 이력만 확인한다.
3. changelog의 `items.field` 값을 기준으로 작업 인정 여부를 판정한다.
4. 아래 필드 변경이 있으면 해당 이슈를 작업 이슈로 인정한다.
   - `status`
   - `assignee`
   - `summary`
   - `description`
   - 시간 추적 관련 필드

### 3. 시간 추적 필드 처리

시간 추적은 Jira changelog에서 실제 어떤 필드명으로 오는지 확인 후 반영한다.

예상 대상은 아래와 같다.

1. `timespent`
2. `timeestimate`
3. `timeoriginalestimate`

실제 응답 기준으로 필요한 필드 목록을 확정한다.

### 4. 댓글 변경 처리 방침

댓글 변경은 이번 단계에서 아래 중 하나로 처리한다.

1. 이번 단계에서는 제외하고 제한사항으로 남긴다.
2. 또는 comment 변경 시각을 별도 확인하는 최소 구현을 추가한다.

단, 댓글 처리는 changelog만으로 완전하게 해결하지 않으므로 범위를 무리하게 넓히지 않는다.

## 설계 원칙

이번 작업은 아래 원칙을 따른다.

1. 사용자 입력 단계는 늘리지 않는다.
2. Query Schema 변경은 최소화한다.
3. 1차 조회와 2차 판정 책임을 분리한다.
4. Jira API 기반으로 처리한다.
5. 기존 `assignee`, `project`, `assignee_project` 세 모드는 유지한다.
6. console/Markdown 출력 구조는 유지하되, 필요하면 작업 판정 기준을 간단히 표시할 수 있다.
7. 대규모 구조 개편 없이 현재 구조 안에서 해결한다.

## 비범위

아래 항목은 이번 작업 범위에서 제외한다.

1. 새로운 CLI 옵션 추가
2. 조회 기준 직접 선택 기능 추가
3. 자연어 기반 조건 해석
4. Web UI 변경
5. MCP 조회 기준 확장
6. 고급 changelog 필터 설정 UI
7. 모든 Jira 활동 종류를 완전하게 포괄하는 범용 활동 엔진 구현

## 권장 구현 방향

구현은 아래 순서를 우선 참고한다.

1. 기존 1차 조회 로직을 유지한다.
2. 1차 후보군에서 설명/요약/시간 추적 변경 이슈가 누락되지 않도록 `updated` 기반 기간 조건을 함께 사용한다.
3. Jira API에서 changelog를 포함해 조회할 수 있는 방식 또는 추가 호출 방식을 검토한다.
4. 기간 내 changelog history만 추린다.
5. `items.field` 값이 추적 대상 목록에 포함되는지 확인한다.
6. 하나라도 일치하면 해당 이슈를 작업 이슈로 인정한다.
7. 댓글은 이번 단계 범위에 맞게 최소 처리 또는 제한사항 처리한다.

## 검증 항목

완료 전 아래 항목을 검증해야 한다.

1. 사용자 입력 단계가 늘어나지 않는다.
2. 상태 변경 이슈가 기간 내 작업 이슈로 잡힌다.
3. 담당자 변경 이슈가 기간 내 작업 이슈로 잡힌다.
4. 설명 또는 요약 변경 이슈가 기간 내 작업 이슈로 잡힌다.
5. 시간 추적 변경 이슈가 기간 내 작업 이슈로 잡힌다.
6. 기간 외 변경 이력은 작업 이슈로 잡히지 않는다.
7. `assignee`, `project`, `assignee_project` 세 모드가 모두 동작한다.
8. `npm run typecheck`가 통과한다.
9. `npm run build`가 통과한다.
10. 가능하면 실제 Jira 데이터 기준으로 변경 전후 결과를 비교한다.

## 주의사항

이번 작업은 “실제 작업 흔적”에 더 가까운 이슈만 보여주기 위한 정밀화 단계다.

다만 Jira changelog와 comment/activity 구조는 완전히 동일하지 않으므로, 댓글 변경이나 일부 앱/자동화 변경은 별도 처리가 필요할 수 있다. 따라서 이번 단계는 우선 changelog 기반 핵심 필드 판정에 집중한다.
